title: Custom Storage
description: Guide on how to implement a custom document storage backend
---

Teleportal was designed to decouple storage from the compute layer, so you can use any storage backend that you want. There are two layers of abstraction when it comes to storage:

- **Layer 1**: `DocumentStorage` - The low-level interface that the server uses to handle raw protocol messages (sync, updates) and store them as they come in.
- **Layer 2**: `AbstractDocumentStorage` - A convenient base class that implements the whole `DocumentStorage` protocol for you (sync-step-1/2, update merging, the content-encryption envelope, and attribution) and delegates only persistence to a small set of methods you implement.

The same `AbstractDocumentStorage` handles both **encrypted** and **unencrypted** documents — there are no longer separate classes. Pass `encrypted: true` (or override the constructor) and the base class stores the plaintext **structure update** alongside the encrypted **sidecars** that hold the document content. For unencrypted documents the sidecars are simply empty and the structure update is the full Y.js update.

## What you implement

When extending `AbstractDocumentStorage`, you only implement persistence for a `DocumentState` (the merged V2 update plus its sidecars) and the document metadata:

```typescript
type DocumentState = {
  update: Uint8Array; // merged Y.js V2 update (CRDT structure, content stripped when encrypted)
  sidecars: IndexedSidecar[]; // encrypted content sidecars (empty for unencrypted docs)
};

abstract class AbstractDocumentStorage {
  getDocumentState(key: string): Promise<DocumentState | null>;
  replaceDocumentState(key: string, update: Uint8Array, sidecars: IndexedSidecar[]): Promise<void>;
  getDocumentMetadata(key: string): Promise<DocumentMetadata>;
  writeDocumentMetadata(key: string, metadata: DocumentMetadata): Promise<void>;
  deleteDocument(key: string): Promise<void>;
  // optional: transaction(key, cb) and storeAttribution(key, attribution)
}
```

The base class takes care of decoding the content-encrypted payload, merging updates, computing state vectors, filtering the relevant sidecars on sync, and writing size/attribution metadata — so your backend never has to understand the wire protocol.

## Example: Custom DocumentStorage

```typescript
import {
  AbstractDocumentStorage,
  type DocumentState,
  type DocumentMetadata,
} from "teleportal/storage";
import type { IndexedSidecar } from "teleportal/protocol/encryption";

// Illustrative in-memory implementation: stores the merged V2 update + sidecars.
export class MyCustomDocumentStorage extends AbstractDocumentStorage {
  private stateMap = new Map<string, DocumentState>();
  private metadataMap = new Map<string, DocumentMetadata>();

  async getDocumentState(key: string): Promise<DocumentState | null> {
    return this.stateMap.get(key) ?? null;
  }

  async replaceDocumentState(
    key: string,
    update: Uint8Array,
    sidecars: IndexedSidecar[],
  ): Promise<void> {
    this.stateMap.set(key, { update, sidecars });
  }

  async writeDocumentMetadata(key: string, metadata: DocumentMetadata): Promise<void> {
    this.metadataMap.set(key, metadata);
  }

  async getDocumentMetadata(key: string): Promise<DocumentMetadata> {
    return (
      this.metadataMap.get(key) ?? {
        createdAt: Date.now(),
        updatedAt: Date.now(),
        encrypted: false,
      }
    );
  }

  async deleteDocument(key: string): Promise<void> {
    this.stateMap.delete(key);
    this.metadataMap.delete(key);
  }
}
```

Wire it into a server like any other storage:

```typescript
const server = new Server({
  storage: new MyCustomDocumentStorage(),
});
```

## Best Practices

1. **Persist state atomically**: `replaceDocumentState` is called with the already-merged update + sidecars — store both together.
2. **Implement `transaction`** if your backend supports it, so concurrent writes to the same document are serialized.
3. **Store metadata separately** from content for efficient listing and size queries.
4. **Handle errors gracefully**: storage operations can fail; let them reject so the server can surface the error.
